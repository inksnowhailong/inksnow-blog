# AI 时代的架构价值：一次重构实验后的反直觉结论

最近我做了一次比较完整的架构重构实验。

其中有一个任务很有意思：我让一个 AI worker 在一次任务里 rename 了项目里的 247 个文件，顺手改掉 100 多处 import 路径，还更新了 ESLint 配置、架构规则文档和 RFC。最后 `tsc --noEmit` 通过，`eslint src/` 也是 0 error。

整个过程里，我没有手动解决冲突，也没有回滚 commit。

这件事乍一看像是 AI 能力很强，但我复盘后反而觉得，真正起作用的不是 prompt，也不是某个神奇模型，而是项目的架构边界足够清楚。

清楚到什么程度？

清楚到 AI 不需要猜，也能知道代码该放在哪里。

这篇文章想聊的就是这个问题：AI 已经会写代码了，学习架构还有价值吗？

我的结论是：有，而且比以前更有价值。只是架构的价值形态变了。

## 一个很常见的误解

现在有一种观点越来越常见：

```text
AI 已经会写代码了。
会描述需求就行。
架构、DDD、Clean Architecture、Hexagonal Architecture 这些东西都过时了。
```

这个判断听起来很合理，因为 AI 确实把“写代码”的门槛打下来了。

但这里有一个隐藏前提：把“能写代码”理解成“知道代码应该放在哪里”。

这两件事其实不是一回事。

让 AI 写一个登录函数，它大概率能写得不错。问题是，这个函数应该放在 `auth/services/`、`utils/`、`api/`，还是 `helpers/`？

在一个小项目里，随便放好像也能跑。

但到了几十个业务模块、几百个文件时，这种“随便找个看起来合理的位置”会不断累积，最后变成架构腐烂。

AI 写代码的能力很强，但 AI 在没有边界约束时，放代码的位置经常是随机的。

而架构的价值，恰恰就是减少这种随机性。

## 我的实验项目长什么样

实验项目是一个中型业务后台，原来是比较典型的 Next.js 扁平结构：

```text
src/
├── app/         Next.js 路由
├── lib/api/     所有数据获取，30+ fetcher
├── types/       业务类型
├── components/  UI 组件
└── stores/      状态管理
```

这种结构早期很舒服，目录少，写起来快。

但业务一多，问题就出来了：数据获取、业务规则、UI 状态、类型定义全部混在一起。AI 要改一个需求时，通常要先扫很多目录，才能拼出“这个业务到底在哪里”。

重构后，我把它改成了更接近 Modular Monolith 的结构：

```text
src/
├── app/         Next.js 薄壳：路由 + UI 组装
├── core/        业务核心，按业务模块垂直切
│   └── {模块}/
│       ├── domain/         业务模型 + 业务规则 + 接口契约
│       ├── application/    use case，编排 domain
│       ├── infrastructure/ Repository 实现 + DB Row mappers
│       └── hooks/          UI 数据流 hook
├── ui/          通用 UI 组件库，0 业务依赖
└── shared/      横切技术底座：supabase/store/auth/format/...
```

核心思想并不复杂：

```text
Modular Monolith
+ 贫血 DDD
+ Ports & Adapters 的最小子集
```

我没有追求教科书式 DDD，也没有把项目拆成微服务。我的目标很实际：让业务模块边界足够清楚，让 AI 和人都能快速判断“这段代码属于哪里”。

## 实验里最明显的变化

重构前，AI 改一个需求经常要来回搜索。它需要看 fetcher、types、components、stores，再自己猜业务主线。

重构后，AI 的定位成本明显下降。因为目录本身就在告诉它：

```text
业务规则去 domain
用例编排去 application
数据库实现去 infrastructure
页面组装去 app
通用 UI 去 ui
横切技术能力去 shared
```

这不是玄学，我记录了一些变化：

| 维度 | 扁平结构 | 模块化分层后 |
|---|---|---|
| AI 定位代码所需 tool 调用 | 通常 5~10 次 | 通常 2~3 次 |
| 单次需求改动文件数 | 5~10 个 | 1~3 个 |
| 单文件最大行数 | 2087 行 | 257 行 |
| 删除一个业务模块 | 散删多个目录 | 删除对应模块目录为主 |
| 换数据源 | fetcher、业务规则、UI 容易混改 | 主要改 infrastructure |
| 业务规则单测 | 很难独立测 | domain 纯函数可以 0 mock |
| AI 放错位置 | 需要人 review 才能发现 | rules + ESLint 可以直接拦截 |
| 大型重构一次通过 | 成功率不稳定 | 实测 6 / 6 一次通过 |
| 跨模块错误依赖 | 很容易悄悄混进去 | `import/no-restricted-paths` 直接红线 |

这些数字不一定能迁移到所有项目，但趋势很明显：架构边界越清楚，AI 越少猜。

对 AI 来说，最贵的不是敲代码，而是判断上下文。

架构做得好，本质上是在减少 AI 的上下文搜索成本和错误决策空间。

如果把这些变化换成一张更直观的图，大概是这种感觉：

```text
            扁平 Next.js                       当前 DDD + 模块化
            ─────────────                      ──────────────────

            AI 找代码    ▓▓▓░░░░░░░            AI 找代码    ▓▓▓▓▓▓▓▓▓░
            改对位置    ▓▓▓▓▓░░░░░             改对位置    ▓▓▓▓▓▓▓▓▓▓
            一次通过    ▓▓▓░░░░░░░             一次通过    ▓▓▓▓▓▓▓▓▓░
            可验证性    ▓▓░░░░░░░░             可验证性    ▓▓▓▓▓▓▓▓▓▓
            弱模型友好  ▓▓░░░░░░░░             弱模型友好  ▓▓▓▓▓▓▓▓▓░
            并行 worker ▓▓░░░░░░░░             并行 worker ▓▓▓▓▓▓▓░░░
            可逆性      ▓▓░░░░░░░░             可逆性      ▓▓▓▓▓▓▓▓▓▓
            启动成本    ▓▓▓▓▓▓▓▓▓░             启动成本    ▓▓▓▓▓▓▓▓░░
            token/单次  ▓▓▓▓▓▓▓░░░             token/单次  ▓▓▓▓▓▓▓▓░░

            综合 AI 友好度：4/10                综合 AI 友好度：9/10
```

这里有两个地方容易误解。

第一，扁平结构的“启动成本”不是低，而是高。因为 AI 新开一个会话后，必须扫多个目录拼业务全貌，看起来目录少，实际理解成本并不低。

第二，`token` 节省不是最大收益。模块化之后文件数会上升，但 AI 单次通常只需要 load 相关层和相关模块，净节省大概是 20% 到 40%。真正大的收益在“少猜”和“少改错”。

如果把它量化成评分，我更愿意这样看：

| 维度 | 评分 | 关键指标 |
|---|---|---|
| 定位速度 | ★★★★★ | AI 定位 token 约 -60% |
| 改动半径 | ★★★★★ | 单次 task 文件数约 -50% |
| 上下文 token | ★★★☆☆ | 净 -20% ~ -40%，但不是最大收益 |
| 生成准确率 | ★★★★★ | 实测 6 / 6 一次成功 |
| 弱模型可用 | ★★★★★ | 最重要的隐性收益之一 |
| 并行 worker 友好 | ★★★★☆ | 偶有 race，但可以靠边界自修复 |
| 可逆性 | ★★★★★ | commit 按 layer / 模块拆，单独 revert 风险低 |
| 可验证性 | ★★★★★ | `tsc` / `eslint` 能给秒级反馈 |
| 启动成本 | ★★★★★ | rules 灌入后，AI 不必全仓库乱扫 |
| 学习代价 | ★★★★★ | AI 时代不再是过去那种纯人力成本 |

综合下来，我会给这套架构形态 **4.6 / 5**。

当然这个分数不是行业基准，只是这个项目里的实验结论。这里最值得关注的不是“DDD 多高级”，而是“弱模型可用”和“启动成本下降”这两项隐性收益。它们直接影响团队能不能用更便宜的模型、更短的会话，完成更复杂的任务。

## 反直觉一：rules 不是文档，是法律条文

过去我们写架构文档，大多是给团队新人看的。

但在 AI 时代，最大的“新人”其实是 AI。

如果规则写成这样：

```text
应当注意单一职责原则。
尽量避免过度耦合。
```

这类话对 AI 没有太强约束力。它会理解，但执行时仍然会摇摆。

更有效的写法是明确禁止项：

```text
禁止：core/*/domain/ 文件 import @supabase/*、react、next/*、fetch、localStorage。

反模式示例：

import { supabase } from '@/shared/supabase'
```

这类规则才像“法律条文”。

它不是表达一种价值观，而是告诉 AI：这里不允许做什么，错了会被工具拦住。

在我的实验里，`.cursor/rules/*.mdc` 配合 `alwaysApply: true`，再加上 ESLint 的 `import/no-restricted-paths`，就形成了一套闭环：

```text
rules 告诉 AI 怎么写
ESLint 告诉系统什么算错
tsc 保证类型契约不乱
```

这才是 AI 时代架构规则真正有用的形态。

写在文档里的建议，约束力很弱。

能被 AI 自动读取、能被 ESLint 拦截、能被 tsc 验证的规则，才会变成工程事实。

## 反直觉二：强架构约束可以降低模型要求

这次实验里，我用的不是最强模型，而是 Cursor 的 generalPurpose subagent。

但 6 次 worker 任务都一次性通过，包括前面提到的 247 个文件 rename，以及把一个 2087 行页面拆成多个更小的模块。

这件事给我的启发很大。

很多人会把 AI 成功率完全归因于模型强弱，但项目约束同样重要。

一个强模型面对弱约束项目，每改一步都要猜：

```text
这个逻辑属于 UI 吗？
属于业务规则吗？
要不要抽到 shared？
能不能直接访问数据库？
这个类型应该放全局还是模块内？
```

一个普通模型面对强约束项目，反而可以按规则执行。

也就是说，架构把一部分“模型推理成本”前置到了规则设计里。

rules 写一次，后面的每个 AI worker 都复用。这个收益是持续的。

从团队成本看，这甚至意味着：你可以用更便宜的模型做更复杂的任务，因为项目结构已经替模型排除了大量错误路径。

## 反直觉三：有些重复代码不是问题

重构时我保留过一处重复代码。

两个业务模块里各有一个 90% 相似的 `getXxxStatusBadge` 函数。按传统 DRY 原则，这种东西很容易被抽到 `shared`。

但我没有这么做。

原因很简单：这两段代码虽然现在相似，但属于不同业务语境。

一旦抽到 `shared`，两个模块就开始共享同一个抽象。以后 compliance 模块想给某个状态加图标，spaces 模块不想加，就会出现抽象层里的条件分支。

那时候你得到的不是复用，而是耦合。

在 AI 时代，这个权衡更明显。

AI 维护两份 10 行相似代码的成本很低，但错误抽象带来的边界污染很贵。

所以我现在更倾向于一个判断：

> 可控重复，有时比错误抽象更健康。

尤其是在业务模块边界还没稳定时，不要为了消灭重复而过早共享。

## 反直觉四：DDD 的旧代价变小了

DDD 和分层架构过去最常被批评的点有两个。

第一，代码量变多。要写 domain、application、infrastructure、mapper、repository 接口，看起来很啰嗦。

第二，学习成本高。新人要理解边界、依赖方向、Ports & Adapters，不如扁平结构上手快。

这些批评在前 AI 时代是成立的。

但 AI 时代，情况有点变化。

代码量增加，对 AI 来说不再是同样意义上的负担。相反，文件拆小后，单文件上下文更小，AI 更容易处理。一个 2000 行文件，对 AI 和人都不友好；十几个 100 到 200 行的文件，反而更适合定位和修改。

学习成本也变了。新人可以直接问 AI：

```text
这个需求应该改哪个模块？
这个逻辑应该放 domain 还是 application？
为什么这里不能 import supabase？
```

只要 rules 写得足够具体，AI 就能引用项目规则给出答案。

这不代表 DDD 没有成本，而是说它的成本结构变了。

过去分层架构的很多样板代码是人手写、人维护、人解释。现在 AI 可以承担大量重复劳动，人更需要负责边界设计和规则制定。

## 架构在 AI 时代的两个载体

这次实验后，我更确定一件事：AI 时代的架构落地，不能只靠口头约定。

我认为至少需要两个载体。

### RFC：记录为什么这么设计

RFC 是给人和 AI 一起读的决策文档。

它不需要每天改，但应该记录清楚：

- 项目现在的问题是什么。
- 为什么要重构。
- 目标目录结构是什么。
- 哪些边界不能破。
- 分阶段怎么落地。
- 验收标准是什么。
- 有哪些风险和代价。

RFC 像“章程”，回答的是方向问题。

它不负责写每条细则，但要记录关键决策，避免半年后大家忘了当初为什么这么做。

### rules：规定每天怎么写

rules 是给 AI 日常执行用的规则。

它应该更具体：

- 每一层允许 import 什么。
- 每一层禁止 import 什么。
- 业务逻辑应该放哪里。
- 反模式代码长什么样。
- 写代码前应该怎么判断位置。

如果 RFC 是章程，rules 就是员工手册。

再往下，就是 ESLint 和 TypeScript：

```text
RFC            决定为什么做
rules          决定怎么做、不能做什么
ESLint + tsc   把规则变成机器约束
```

这三层配合起来，架构才不是“写在文档里的愿望”，而是每天写代码时会真实生效的系统。

## 这套方法不适合所有项目

我不想把这件事讲成万能方法。

如果一个项目只有两三个业务模块，强行搞完整 DDD，大概率就是噪音。一个普通 to-do list 应用，不需要 domain/application/infrastructure 分那么细。

我现在比较粗暴的判断是：

```text
业务模块 <= 3 个：优先保持简单
业务模块 >= 4 个：开始考虑按业务垂直切
```

另外，这次实验的成功不是“分层”单独带来的，而是一套组合拳：

- 分层让 AI 知道代码大概放哪。
- rules 把边界写成 AI 可读规则。
- ESLint 拦截跨层错误依赖。
- tsc 保证类型契约。
- 显式 barrel export 降低跨模块深路径 import。

只做其中一项，收益会打折。

还有一点也要诚实说：这毕竟是一个项目里的实验，不是行业普查。不同业务复杂度、不同框架、不同团队习惯，得到的结果可能不一样。

所以更准确的说法不是“所有项目都该这样”，而是：

> 当项目进入中等复杂度后，架构边界会直接影响 AI 协作效率。

## 我把方法做成了一个 skill

这次实验后，我把其中一部分方法沉淀成了一个 Cursor skill：

```text
tvs-inksnow-arch
```

两个地址都可以拉取，是同一份代码、独立托管，按你常用的平台二选一：

- GitHub：<https://github.com/inksnowhailong/ai-tools-skills/tree/23fc3395cf0b2a8d0a7b00d32b318a7037c8c033/skills/tvs-inksnow-arch>
- ClawHub：<https://clawhub.ai/inksnowhailong/tvs-inksnow-arch>

它的目标不是替你“自动重构整个项目”，而是先把架构决策问清楚，再产出 RFC 和 rules。

大概流程是：

```text
表达架构重构需求
  ↓
访谈确认核心决策
  ↓
生成架构 RFC
  ↓
等待确认
  ↓
扫描项目结构
  ↓
生成 .cursor/rules/*.mdc
```

这里我特别强调“访谈确认”，因为架构不是纯代码问题。

AI 可以帮你写规则，但规则背后必须有人判断：业务模块怎么切、数据源是否需要可替换、是否适合一次性重构、哪些边界是必须守住的。

这些问题不问清楚，直接让 AI 改目录，风险很高。

## 最后：AI 时代架构师的位置上移了

回到开头的问题：AI 会写代码之后，学习架构还有价值吗？

我的答案是：更有价值。

但架构师的工作位置变了。

以前架构师经常要写示范代码、做培训 PPT、在 code review 里反复解释“这里不该这么写”。

AI 时代，这些事情仍然存在，但更核心的产物变成了：

```text
给人和 AI 共读的决策文档
给 AI 自动执行的强约束规则
能被工具验证的边界
```

也就是说，架构师不再只是写更漂亮的代码，而是要把业务事实、技术机制、平台细节、UI 表现之间的边界划清楚。

AI 能根据规则写代码。

但 AI 很难自己制定一套好规则。

因为好规则不是从代码里凭空长出来的，它来自业务判断、演进预判、团队成本和技术约束之间的平衡。

这正是架构能力的价值。

AI 让写代码变便宜了，但也让“把代码放在正确的位置”变得更重要。

代码越容易生成，边界越需要被设计。

这就是我这次实验得到的最大结论。
