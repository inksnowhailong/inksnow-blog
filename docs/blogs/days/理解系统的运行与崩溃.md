# 从一次请求到 AI Agent：系统如何避免“自我爆炸”

## 一、问题起点：一个请求到底经历了什么？

在讨论 AI Agent 之前，必须先回到最基础的问题：**一个用户请求进入系统后，发生了什么？**

一个典型 Web 请求链路如下：

```text
浏览器
→ DNS 解析
→ TCP / TLS 建立连接
→ CDN（缓存层）
→ 网关（Nginx / Gateway）
→ 应用服务（业务逻辑）
→ 缓存（Redis）
→ 数据库（MySQL）
→ 返回响应
```

这条链路的关键在于：**每一层都在“减少压力”或“控制风险”**。

* CDN：拦截静态请求，减少后端压力
* 网关：做限流、鉴权、防止非法请求进入
* 应用层：执行业务逻辑
* 缓存层：减少数据库访问
* 数据库：最终数据来源（也是最脆弱的一层）

这套体系本质上是在解决一个问题：

> **如何在高并发下，保证系统不崩溃**

---

## 二、高并发系统是怎么崩的？

很多人直觉认为：系统是被“流量太大”打崩的。

这是错的。

更准确的说法是：

> **系统是被“放大效应”打崩的**

来看一个典型崩溃链路：

```text
缓存过期（热点数据）
→ Redis 未命中
→ 请求打到数据库
→ 数据库变慢（慢查询 / 锁）
→ 连接池耗尽
→ 应用线程阻塞
→ 请求堆积
→ 网关超时
→ 全站崩溃
```

注意这里的关键：

* 初始流量可能并不大
* 但每一层都会**放大问题**

例如：

* 缓存 miss → 放大 DB 压力
* 慢查询 → 放大线程占用时间
* 重试机制 → 放大请求数量

最终形成一个典型特征：

> **问题会被“层层放大”，最终失控**

---

## 三、核心认知：系统不是抗流量，而是抗“放大”

这一点非常重要。

传统系统设计的本质不是：

> 提高系统能承受多少 QPS

而是：

> **控制每一层不要放大问题**

---

## 四、AI Agent 带来的本质变化

现在我们把视角切到 AI 系统。

在没有 Agent 的情况下：

```text
用户请求 → 系统处理 → 返回结果
```

这是一个**线性模型**。

---

但引入 Agent 后：

```text
用户请求
→ LLM 分析
→ 决定调用工具
→ 工具返回
→ 再次分析
→ 再生成任务
→ 再调用工具
```

关键变化在这里：

> **系统开始“自己生成任务”**

---

## 五、问题升级：从“放大效应”到“指数爆炸”

在 AI Agent 系统中，问题不再是“某一层放大”，而是：

> **整个调用链开始指数增长**

用一个简单模型表示：

```text
总调用量 ≈ 分支数 ^ 深度
```

举例：

* 每一步调用 2 个工具，深度 3 → 8 次调用
* 每一步调用 3 个工具，深度 4 → 81 次调用
* 每一步调用 5 个工具，深度 5 → 3125 次调用

这意味着：

> **系统可能在极短时间内自我膨胀到无法承受的规模**

---

## 六、关键本质：系统在“自我生成负载”

传统系统：

> 负载来自用户

AI 系统：

> 负载来自系统内部

这就是最本质的变化。

---

## 七、错误直觉：限制调用次数就够了？

很多人会这么想：

> 给 Agent 加一个 maxSteps，不就解决了吗？

这是不够的。

因为真正的问题不是“次数”，而是：

* 是否递归（深度）
* 是否并发扩散（分支）
* 是否有重试放大

---

## 八、正确解法：控制增长模型

要解决这个问题，必须控制三个维度：

---

### 1. 深度（Depth）

限制递归层级，避免无限调用：

```text
Agent → Agent → Agent（最多 3 层）
```

---

### 2. 分支（Branching）

限制每一步的扩展能力：

```text
一次最多调用 1~2 个工具
```

---

### 3. 总量（Budget）

限制整体资源消耗：

* 最大步骤数
* 最大 token
* 最大成本

---

这三者组合，才能真正控制系统：

> **不是限制行为，而是限制增长速度**

---

## 九、架构升级：必须引入“控制层”

为了实现上述控制，需要一个新的架构分层：

```text
用户请求
↓
Router（是否需要 Agent）
↓
Agent Runtime（控制层）
↓
LLM（负责决策）
↓
Scheduler（控制执行）
↓
Tools（执行）
↓
返回结果
```

---

## 十、关键设计思想：AI 不直接控制系统

这是整个架构中最重要的一点。

---

### ❌ 错误方式

让 LLM 决定：

* 是否执行
* 并发多少
* 调用多少工具

---

### ✅ 正确方式

```text
LLM：提出建议（调用什么）
系统：决定执行（是否允许）
```

---

换句话说：

> **AI 负责“想”，系统负责“控制”**

---

## 十一、Scheduler 的角色

Scheduler 是整个系统的“刹车系统”。

它负责：

* 控制并发（max concurrency）
* 控制执行顺序（优先级）
* 控制任务队列

例如：

* 用户请求优先级最高
* Agent 子任务优先级降低（深度惩罚）
* 长时间等待的任务逐渐提升优先级

---

## 十二、Runtime 的角色

Runtime 是“总控层”，负责：

* 记录执行步骤
* 控制预算（steps / cost / tokens）
* 判断是否终止流程
* 防止死循环

它的核心职责是：

> **随时可以终止系统行为**

---

## 十三、Router 的角色

Router 用来判断：

> 当前请求是否需要进入 Agent 模式

例如：

* 简单问答 → 直接 LLM
* 查询 + 分析 → Agent
* 复杂任务 → 多步骤 Agent

这一步可以用 AI 辅助，但必须有规则兜底。

---

## 十四、整体抽象：确定性包裹非确定性

整个系统可以抽象为：

```text
Deterministic System（可控）
    ↓
AI Decision（不可控）
    ↓
Execution（受控执行）
```

核心思想：

> **用确定性系统包住非确定性 AI**

---

## 十五、最终结论

我们可以把整个问题统一起来：

---

### 1. 传统系统

问题：高并发
解法：控制放大效应

---

### 2. AI Agent 系统

问题：自我膨胀
解法：控制增长模型

---

### 3. 核心差异

| 系统类型  | 核心风险 |
| ----- | ---- |
| 传统系统  | 流量过大 |
| AI 系统 | 内部爆炸 |
