这一年我一直在折腾一件事：让多个独立的 AI chat 像一个团队一样协作。

起因很现实。像 Cursor 这类工具，一个 chat 就是一个独立的 agent，有自己的上下文、自己的进程，但它们之间是**彼此隔绝的孤岛**——没有原生的多 agent 编排，A 不知道 B 在干什么，更别说互相派活、共享结论。我想要的是一个能"派任务、回结果、共享上下文、还能在消息到来时自动醒过来"的团队系统。

最省事的做法当然是上一个后端：消息队列、共享数据库、WebSocket 长连。但我给自己定了个苛刻的约束——**零后端、零外部依赖，只用文件系统 + 编辑器自带的 hook 机制把这事干成**。一来是不想让任何人为了用它去装一堆环境，二来我隐约觉得，越是被约束逼出来的设计，越能暴露问题的本质。

这篇就讲讲我最后定下来的设计，以及每个决定背后的取舍。不贴代码，只讲思路和实现机理。

我选了**星型**而不是网状：一个 leader + N 个 sub，所有通信都经 leader 中转，sub 之间不直接说话。

这个决定看起来"中心化得不够优雅"，但它换来一个极其重要的性质：**协调复杂度从网状的 O(N²) 降到了 O(N)**。leader 是唯一的编排点，也是唯一的上下文汇聚点。任何一次"谁该干什么、谁的结论是什么"都有唯一的裁决者，调试和归因都变得简单。代价是 leader 会成为瓶颈，所以我把 sub 数量控制在 3~4 个最舒服，超过七八个就该考虑拆成两个团队了。

另一个我从一开始就钉死的规矩：**任何代码型产出都默认走"实现者 → 审查者"两道手**。团队里必须至少有一个审查角色。这是为了从流程上堵掉"自己写自己审"的盲区——一个人(或一个 agent)永远看不见自己的某些错误，这点对 AI 尤其成立。

## 通信：把文件系统当成邮箱

agent 之间怎么传消息？我的答案是 **per-sender 的文件邮箱**。

每个接收者有一个收件箱目录，目录里**再按发送者分子目录**；每一条消息是一个独立的 JSON 文件，文件名带时间戳。发消息就是往对方收件箱里对应自己的那个子目录丢一个文件，收消息就是把自己收件箱里的文件全读出来、读一条删一条。

听起来很土，但这套土办法里藏着三个让它"够用且健壮"的关键决定：

**第一，写入必须是原子的。** 直接往目标文件写，读取方可能读到写了一半的内容。我的做法是先写一个临时文件，写完再 rename 顶上去——在同一个文件系统里 rename 是原子操作，所以读取方要么看到旧的(或没有)，要么看到完整的新文件，**永远不会读到半截 JSON**。这一条是整套设计能不加锁的地基。

**第二，按发送者分目录，从物理上消灭写冲突。** 既然每个发送者只往"属于自己的"子目录里写，那就根本不存在两个发送者抢同一个文件的情况。**不需要任何锁**——并发安全不是靠控制并发实现的，是靠让冲突不可能发生实现的。

**第三，读即删，天然 at-most-once。** 消息被消费后立刻从磁盘删除，所以同一条消息不会被处理两次，也不用维护"已读/未读"状态。目录自己就是队列，空了就是处理完了。

为什么不用现成的消息队列？因为这套方案有三个它们给不了的好处：**零依赖**(一个文件系统而已)、**完全可观测**(消息就是文件，出问题直接去目录里看)、**崩溃可恢复**(没处理完的消息还躺在文件里，重启接着读)。对一个跑在本地编辑器里的协作系统来说，这三点比吞吐量重要得多。

## 共享上下文：用"权限"代替"锁"

团队需要一块公共记忆——当前在干什么、定了哪些决策、有哪些约定。我叫它**黑板**。

最容易踩的坑是：多个 agent 同时写黑板，于是又要回到锁、版本协商、甚至 CRDT 那一套并发控制里去。我绕开了整个问题，靠一个约束：**只有 leader 能写黑板，所有 sub 只读。**

单写者，意味着**并发写从根上不可能发生**。共享可变状态最难的部分——写竞态——直接被一条权限规则消灭了。这是我越来越喜欢的一种思路：**很多并发问题，与其用更复杂的机制去"控制"，不如用更简单的约束让它"不发生"。** 约束比能力更值钱。

黑板内部我还做了一点区分：**决策**用只增不改的追加日志记录(一条决策一行，历史天然可追溯)，**上下文**和**约定**用整文件覆盖。该留痕的留痕，该刷新的刷新。

顺带一提，我把团队要用的"知识"刻意分了三层、互不重叠：**代码结构**类的问题(谁定义了它、谁调用了它)交给 AST 知识图谱工具，机器维护、秒级新鲜；**业务领域知识**放专门的记忆库；**本次协作的临时状态**才放黑板。三层各管一摊，agent 查什么去对应的层，既省 token 又更准。把知识按"谁来维护、多久过期"分类，是我觉得很多记忆系统没想清楚的地方。

## 事件唤醒：让被动的 agent 自己醒过来

这是整套设计里我最得意的一块，也是最难的一块。

问题是这样的：一个 chat/agent 本质上是"请求-响应"的——你说一句，它答一句，然后就**静止**了。它没有事件循环，不会"有消息进来就自动醒过来处理"。可一个团队成员如果不能被动唤醒，那"协作"就无从谈起——leader 派了活，sub 得能醒过来接。

在一个没有事件循环的宿主里，怎么造出"事件驱动"？

我盯上了编辑器的一个机制：**每次 agent 停下来时，会触发一个 hook**。于是我让这个 hook 在每次停止时做一件事——检查自己的收件箱有没有积压。有，就让 chat 去处理；没有，就让 chat **在后台启动一个监听进程**，然后安心停下。

这个后台监听进程，就是整个机关的核心。它盯着收件箱，**在有新消息、或空闲超时(我设的 60 分钟)时，自己退出**。

而真正巧的地方在这里——

> **编辑器会把"你启动的后台进程结束了"当作一个『任务完成通知』，从而再次唤醒这个 chat。**

于是我**故意让这个监听进程的『退出』，去承载『有新消息』这个事件**。进程一退出，chat 被叫醒，叫醒后那个停止 hook 又跑一遍，这次发现收件箱有货了，就让 chat 去处理。处理完又停下，又进入待命……一个完整的闭环就此跑起来。

说白了，我是**借宿主自带的"进程结束通知"，反向拼出了一个它本身并不提供的 event-driven 唤醒机制**。等于白嫖了一个事件总线。当我第一次让这套闭环跑起来、看着一个本该静止的 chat 真的自己醒过来接活时，那种"用现有积木拼出了原本没有的能力"的快感，是我做这套东西最大的回报。

当然，凡是这种"自己唤醒自己"的设计，都要防一件事：**死循环**。停止→唤醒→又停止→又唤醒……所以我加了一道闸：如果监听进程已经在跑了，就别再重复唤醒——靠探测那个后台进程是否存活来判断。有的编辑器自带"同一次停止最多续一次"的上限，那就更省事，直接用它兜底。

## 边界与取舍

我不想把它吹成银弹，它的代价很清楚：

- **慢**。文件邮箱靠文件事件 + 轮询兜底，延迟是秒级的，不适合高频通信。但团队协作本来就不是高频场景，可以接受。
- **强绑 hook 协议**。整套唤醒机制依赖编辑器的停止 hook 行为，每个工具的协议都不一样，要分别适配。这也是它最脆的地方——**一旦宿主自己推出了原生多 agent 编排，这套就会被取代**。我对此很清醒：它是一个"在平台补上这块能力之前"的填空方案，不是长期基础设施。
- **可移植性的命门，是会话标识**。整个唤醒靠"当前这个 chat 是哪个成员"的映射成立，而这个映射依赖停止 hook 能给出一个稳定的会话 ID。给得出，整套能搬；给不出，邮箱和黑板还能用，但自动唤醒就得另想办法。所以谁想借鉴它，第一件事不是看通信，是去翻自己那个宿主的 hook 到底给不给会话标识。

---

## 写在最后

这套东西真正让我有收获的，不是"我做了个多 agent 系统"，而是它逼我反复回到几个很朴素的原语上：**文件、进程、权限**。

- 用**原子 rename**，让文件系统当上了无锁消息队列；
- 用**单写权限**，让共享状态绕开了整个并发控制；
- 用**进程的生命周期**，在没有事件循环的地方造出了事件驱动。

没有一样是新东西，但把它们摆在对的位置上，就拼出了一个原本需要后端才能干的事。我越来越相信，**好的设计往往不是引入更强的机制，而是找到那个让复杂性自己消失的约束。** 这套小小的团队系统，是我对这句话的一次实践。
